# cassette — agent orientation

This file is for agents working on the repo. Read it before editing.

## What this project is

`cassette` is a self-hosted, YouTube-shaped personal video platform. Originals are uploaded over a simple HTTP API
(channel-scoped Better-Auth API keys), pre-transcoded by a pg-boss worker into an HLS ABR ladder with MPEG-TS `.ts`
segments, and served back over Range-supporting Next.js route handlers. The frontend is publicly accessible: anonymous
viewers can watch any public or unlisted video without an account; auth is required for uploading, liking, commenting,
subscribing, playlists, history, and the queue.

The full design lives in `PLAN.md`. The live execution checklist lives in `TODO.md`. Do not duplicate either of those
into this file. Check them whenever you're unsure.

## Stack rules

- Package manager: **yarn**. Never run `npm` or `pnpm`. `corepack enable` first.
- Task runner: **just**. `Justfile` at repo root.
- Database access: **Drizzle**. No raw SQL except in `triggers.sql` and migrations.
- Auth: **Better-Auth**. Schema is generated by Better-Auth's CLI; do not hand-edit the auth tables.
- Routing: **tRPC v11** for typed procedures, plus REST handlers for the upload (binary) and HLS endpoints.
- Player: **Vidstack** with a custom layout. Do not use `DefaultVideoLayout`.
- Components: **shadcn/ui** primitives, restyled in `src/components/ui/*`. Do not import Radix directly in feature code.
- Typography: `system-ui` only. No Google Fonts. No custom font binaries.

## Code style (strict)

- 4-space indent. 120-col print width. Semicolons. Double quotes. Trailing commas.
- ESM only. Node 22 target. Strict TypeScript. No unannotated `any`.
- Named exports unless the file is a Next.js entry (`page.tsx`, `layout.tsx`, `route.ts`).
- Prefer `node:` imports for built-ins.
- Functional style with `async/await`. Minimise classes.
- Guard clauses and early returns. Destructuring where it helps.
- Comments explain *why* and call out trade-offs. They do not narrate the next line.
- ASCII-clean code, comments, and technical docs. No fancy unicode dividers.

Run `yarn format` and `yarn lint` before committing.

## File layout (high level)

```
src/
    app/                  Next.js App Router (pages, layouts, route handlers)
    components/           Feature components and shadcn primitives (ui/)
    lib/                  Cross-cutting helpers (auth, hls/, transcode/, paths, slug, utils)
    server/
        api/              tRPC routers and context
        db/               Drizzle client and schema (per-domain files)
    worker/               pg-boss boot + jobs
docker/                   Dockerfile and DB init SQL
drizzle/                  generated migrations (do not hand-edit)
scripts/                  one-shot scripts (seed, migrate, smoke)
```

## Conventions

- **One paths module**: `src/lib/paths.ts` resolves `MEDIA_SOURCE_PATH` and `MEDIA_HLS_PATH`. Do not duplicate these
  resolutions across the codebase.
- **One env module**: `src/env.ts` validates and exposes env. Import from there, not `process.env`.
- **Single source of truth for tRPC procedures**: `src/server/api/routers/<domain>.ts`.
- **No back-compat shims** while we are pre-1.0. If something is renamed, update all callers in the same commit.

## Database

- Postgres extensions used: `citext`, `pg_trgm`, `pgcrypto`. Loaded automatically in dev by
  `docker/init-extensions.sql`. In production they need to be created once.
- Schema files are split per domain under `src/server/db/schema/`. The `index.ts` in that folder re-exports them.
- Use `yarn db:push` while iterating; switch to `yarn db:generate` + a migration once the schema stabilises.

## Commits

- Conventional Commits subjects (`feat(scope): ...`, `fix(scope): ...`, `chore(scope): ...`).
- Keep the body informative. Document intent, important decisions, validations run, and follow-ups.
- Commit continuously as meaningful slices land. Do not batch the whole feature into one commit.
- Keep the tree green. Do not commit a tree that does not build.
